Robot Modelling and Simulation base on ROS

Introduction

The current carbon fiber reinforced polymers(CFRP) manufacturing process is labor intensive, especially in the vacuum bagging stage. In manufacturing CFRP components in the vacuum infusion process, one of the main steps is vacuum bagging. Vacuum bagging is an intricate process that involves tightly layering the thin plastic film over a mold, forming and positioning pleats into the small radius features, and smoothing wrinkles across the surface in order to mitigate defects such as voids or incomplete lamination. The current process is labor intensive and when manufacturing large parts, the technician must work from an overhang which strains the worker. Therefore, to improve the process, a proof of concept robot arm end-effector which can semi-automate the vacuum bagging process while minimizing defects was the objective for the project.

Related Work

Pleating Protocol

Pleating protocol is a vacuum bagging protocol for a roller-based pleat manipulation. When manipulating pleat, there are several challenges such as pleat straightness, multi-folds, or offset from the desired location. Therefore, special actions are required to resolve these issues. There are 4 steps in the proposed pleating protocol. First, partial vacuum of around 3 mbar is applied. Next, the end-effector will smooth the surface by pushing all the wrinkles towards the existing pleats. Further, the two sides of pleat are rolled in opposite direction so that pleat can be moved to the desired position. Lastly, the end-effector is lifted vertically to pull the pleat into a taut fold.

End-effector

A proof of concept robot end-effector have be designed featuring a pair of ultrasonic sensors, accelerometer, force sensing resistors, Wi-Fi module, motor current sensors, potentiometer, and an Arduino micro controller. The end-effector inherits the form of a spring-loaded jaw which meets at the tip with high friction silicon elastomer rollers for pleat manipulation. The jaw is opened and closed via a servo. The torsional springs were selected according to the friction coefficient between the rollers and bagging material.

Robot Modeling and Simulation

A robotics simulator is used to create application for a physical robot without depending on the actual machine, thus saving cost and time. In some case, these applications can be transferred onto the physical robot (or rebuilt) without modifications.

The term robotics simulator can refer to several different robotics simulation applications. For example, in mobile robotics applications, behavior-based robotics simulators allow users to create simple worlds of rigid objects and light sources and to program robots to interact with these worlds. Behavior-based simulation allows for actions that are more biological in nature when compared to simulators that are more binary, or computational. In addition, behavior-based simulators may “learn” from mistakes and are capable of demonstrating the anthropomorphic quality of tenacity.

In this project, We use two connected robot arms instead of humans. The handler who wears HoloLens can set the goal points in 3D space. Once the arms receive the target, computer could obtain inverse kinematic solutions, and the end-effector will arrive all the positions one by one.

Physical Simulation and Model Controlling

This is an overview of robot modeling and simulation. Firstly, we need to use unified robot description format to describe our robot. Secondly, we could import Solidworks file into URDF, and load this URDF at physical simulation environment, one of the most popular is Gazebo. Although we have established a robot model in the physical simulation environment, we need to control it. ROS control is a robot controlling frame, includes hardware interface and controller manager. Controller manager manage many controller which can send commands to hardware and receive robot state message. In the one hand, ROS control can provide interface to users. In another hand, ROS control can communicate with hardware or simulation environment.

![](D:\Hank\UBC-Summer\Mywork\Daily Report\Report\summer report\Fig8.PNG)

Simulating a robot’s controllers in Gazebo can be accomplished using ros_control and a simple Gazebo plugin adapter. An overview of the relationship between simulation, hardware, controllers and transmissions is shown below:

![](D:\Hank\UBC-Summer\Mywork\Daily Report\Report\summer report\Fig9.PNG)

To use ros_control with your robot, you need to add some additional elements to your URDF. The <transmission> element is used to link actuators to joints, see the spec for exact XML format.

For the purposes of gazebo_ros_control in its current implementation, the only important information in these transmission tags are:

  • <joint name=""> - the name must correspond to a joint else where in your URDF
  • <type> - the type of transmission. Currently only “transmission_interface/SimpleTransmission” is implemented. (feel free to add more)
  • <hardwareInterface> - within the <actuator> and <joint> tags, this tells the gazebo_ros_control plugin what hardware interface to load (position, velocity or effort interfaces). Currently only effort interfaces are implemented. (feel free to add more)

The rest of the names and elements are currently ignored.

In addition to the transmission tags, a Gazebo plugin needs to be added to your URDF that actually parses the transmission tags and loads the appropriate hardware interfaces and controller manager. By default the gazebo_ros_control plugin is very simple, though it is also extensible via an additional plugin architecture to allow power users to create their own custom robot hardware interfaces between ros_control and Gazebo.

The default plugin XML should be added to your URDF:

The gazebo_ros_control <plugin> tag also has the following optional child elements:

  • <robotNamespace>: The ROS namespace to be used for this instance of the plugin, defaults to robot name in URDF/SDF
  • <controlPeriod>: The period of the controller update (in seconds), defaults to Gazebo’s period
  • <robotParam>: The location of the robot_description (URDF) on the parameter server, defaults to ‘/robot_description’
  • <robotSimType>: The pluginlib name of a custom robot sim interface to be used (see below for more details), defaults to ‘DefaultRobotHWSim’

By default, without a <robotSimType> tag, gazebo_ros_control will attempt to get all of the information it needs to interface with a ros_control-based controller out of the URDF. This is sufficient for most cases, and good for at least getting started.

The default behavior provides the following ros_control interfaces:

  • hardware_interface::JointStateInterface
  • hardware_interface::EffortJointInterface
  • hardware_interface::VelocityJointInterface - not fully implemented

The gazebo_ros_control Gazebo plugin also provides a pluginlib-based interface to implement custom interfaces between Gazebo and ros_control for simulating more complex mechanisms (nonlinear springs, linkages, etc).

These plugins must inherit gazebo_ros_control::RobotHWSim which implements a simulated ros_control hardware_interface::RobotHW. RobotHWSim provides API-level access to read and command joint properties in the Gazebo simulator.

User Interface based on MoveIt!

User interface is based on MoveIt! interface. In this project, we use MoveIt! setup assistant to generate a demo. And according to the Python interface document, we written a script as the user interface.

One of the simplest MoveIt! user interfaces is through the Python-based Move Group Interface. These wrappers provide functionality for most operations that the average user will likely need, specifically setting joint or pose goals, creating motion plans, moving the robot, adding objects into the environment and attaching/detaching objects from the robot.

There are 8 commands (numbers) in the list. All the commands are orginazied in switch case, like that

Command List

0 : stop DC motors

1 : drive DC motors in clockwise from the servo direction

2 : drive DC motors in anticlockwise from the servo direction

3 : motor B enable debug

4 : motor A enable debug

5 : disable motors debug

6 : scan motor A I2C bus

8 : open servo (open the gripper)

9 : close servo (close the gripper)
